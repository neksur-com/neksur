# Runbook: Customer-Facing Dremio Deployment (Neksur Policy Gateway)

**Owner:** Customer Engineering / Platform Engineering
**Scope:** Configure a customer's Dremio instance to route Iceberg REST catalog
traffic through the Neksur L1 gateway as the upstream catalog endpoint. This
runbook covers Dremio version compatibility, the `iceberg.rest-catalog.uri`
configuration, query smoke tests, and the Phase 3 `nightly-cross-engine.yml`
PENDING_FIRST_RUN gate for the Dremio leg.
**Closes:** 03-VALIDATION.md Manual-Only §Per-engine SQL probe semantics (Dremio
leg); Phase 3 Plan 03-05 (live Dremio dialect).

---

## 1. Prerequisites

| Item | Required | How to verify |
|------|----------|---------------|
| **Dremio** | OSS 24.0+ or Enterprise | `dremio --version` or Admin UI → About |
| **Neksur gateway** | Running + reachable from Dremio host | `curl https://neksur-gateway:8080/api/` returns 401 (auth gate) |
| **Iceberg REST endpoint** | Enabled on the Neksur gateway | `curl https://neksur-gateway:8080/v1/config` returns 200 |
| **Tenant ID** | Provisioned in Neksur | `neksur-cli tenant list` |
| **mTLS certs** | Client cert + key + CA bundle for Dremio | `ls /etc/neksur/dremio-client.{crt,key,ca}` |

---

## 2. Neksur Gateway Endpoint Configuration

The Neksur L1 catalog gateway exposes an Iceberg REST catalog at:

```
https://<neksur-gateway-host>:<port>/v1/
```

This is the value you will configure as `iceberg.rest-catalog.uri` in Dremio.

Dremio speaks to the Neksur gateway as an Iceberg REST catalog client — from the
gateway's perspective, Dremio is identical to Trino or Spark (per D-3.02). The
same row-filter + column-mask enforcement that applies to Trino SELECT queries
applies to Dremio queries via the Phase 3 live Dremio dialect (Plan 03-05).

## 3. Dremio Version Compatibility

| Dremio Version | Iceberg REST support | Notes |
|----------------|---------------------|-------|
| 23.x | Partial (catalog integration experimental) | Not supported by Phase 3 Neksur adapter |
| **24.0+ (OSS)** | **Full — use this** | Tested against Neksur Phase 3 testcontainer |
| 25.0+ | Full | Tested; also the `dremio-oss:latest` testcontainer version |
| Enterprise 24.2+ | Full | Same REST implementation; additional connector features irrelevant to Neksur |

**Phase 3 CI testcontainer uses `dremio/dremio-oss:25.0` (see `tests/testfixture/dremio.go`).**
Use Dremio 24.0+ in production.

## 4. Dremio Source Configuration

### 4.1 Configure via Dremio Admin UI

1. Open Dremio Admin UI → **Sources** → **Add Source**.
2. Select **Nessie** (Dremio uses Nessie-compatible Iceberg REST for external catalogs).
   If your Dremio version shows "Iceberg REST" directly, use that.
3. Configure:

| Field | Value |
|-------|-------|
| **Name** | `neksur_catalog` (or customer-chosen name) |
| **Nessie/Iceberg REST endpoint** | `https://<neksur-gateway-host>:<port>/v1` |
| **Authentication** | Bearer token (Neksur issues short-lived tokens per D-2.09) |
| **Token** | Set `NEKSUR_DREMIO_TOKEN` env var — do NOT hardcode |
| **TLS** | Enable; provide CA bundle path |

### 4.2 Configure via Dremio REST API (automation-friendly)

```bash
curl -X POST https://dremio-host:9047/api/v3/source \
  -H "Authorization: Bearer $DREMIO_ADMIN_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "entityType": "source",
    "name": "neksur_catalog",
    "type": "NESSIE",
    "config": {
      "nessieEndpoint": "https://<neksur-gateway-host>:<port>/v1",
      "nessieAuthType": "BEARER",
      "bearerToken": "'"$NEKSUR_DREMIO_TOKEN"'",
      "awsRootPath": "/",
      "credentialType": "NONE",
      "secure": true
    }
  }'
```

### 4.3 Iceberg REST direct configuration (Dremio 25.0+ preferred)

```properties
# In dremio.conf or environment override:
iceberg.rest-catalog.uri=https://<neksur-gateway-host>:<port>/v1
iceberg.rest-catalog.type=rest
iceberg.rest-catalog.credential=$NEKSUR_DREMIO_TOKEN
```

## 5. mTLS Configuration (Internal Engines)

If Dremio is behind the Neksur SQL proxy (internal engine — mTLS required per
Phase 02 D-2.08), the Dremio process needs client certificates:

```bash
# Place the cert material (generated by ACM PCA per Plan 02-08):
/etc/neksur/dremio-client.crt   # Client certificate (PEM)
/etc/neksur/dremio-client.key   # Client private key (PEM) — chmod 600
/etc/neksur/neksur-ca.crt       # Neksur CA bundle (PEM)

# Configure in Dremio:
iceberg.rest-catalog.transport.ssl.client.cert=/etc/neksur/dremio-client.crt
iceberg.rest-catalog.transport.ssl.client.key=/etc/neksur/dremio-client.key
iceberg.rest-catalog.transport.ssl.ca.bundle=/etc/neksur/neksur-ca.crt
```

## 6. Query Smoke Tests

After configuration, run these smoke tests to confirm end-to-end policy enforcement:

### 6.1 Table discovery

```sql
-- In Dremio SQL editor:
SHOW TABLES IN neksur_catalog.prod;
-- Expected: list of tables registered in Neksur, including row-filter-protected ones.
```

### 6.2 Row-filter enforcement

```sql
-- Query a table that has a row-filter policy (e.g., deleted_flag=false):
SELECT count(*) FROM neksur_catalog.prod.orders;
-- Expected: count of non-deleted rows (policy enforced by Neksur gateway).

-- Without policy enforcement, this would return all rows.
-- Verify by comparing against a direct Polaris query with no policy.
```

### 6.3 Column-mask enforcement

```sql
-- Query a table with a column-mask policy (e.g., SSN masked):
SELECT ssn FROM neksur_catalog.prod.customers LIMIT 5;
-- Expected: SSN values masked to 'XXX-XX-LAST4' format.
```

### 6.4 Write-coordinator enforcement (Dremio INSERT)

```sql
-- Attempt an INSERT that violates a policy (e.g., missing required tag):
INSERT INTO neksur_catalog.prod.customers VALUES (...);
-- Expected: 403 response with policy_class + policy_id in Dremio error output.
-- (D-3.03 write-coordinator pre-commit hook enforcement — Plan 03-09)
```

## 7. PENDING_FIRST_RUN Gate

Per 03-ACCEPTANCE.md §9, the "Dremio testcontainer integration tests" acceptance row
is `PENDING_FIRST_RUN` until the nightly-cross-engine.yml workflow exits 0 with
Dremio testcontainer tests passing. This runbook covers the manual operator smoke-test
required alongside automated CI.

Steps to flip the Dremio PENDING_FIRST_RUN row:
1. Confirm `nightly-cross-engine.yml` exits 0 with the Dremio integration job passing.
2. Run the §6 smoke tests against the staging cluster.
3. Record the result in 03-ACCEPTANCE.md §9 Dremio row: `PASS — nightly CI exit-0 + operator smoke test <date>`.

---

## 8. Pass / Fail Checklist

| # | Check | Pass Criterion |
|---|-------|----------------|
| 1 | Dremio source created | `SHOW TABLES IN neksur_catalog.prod` returns tables |
| 2 | Row-filter enforced | `SELECT count(*)` returns filtered count (not total) |
| 3 | Column-mask enforced | SSN column returns masked values |
| 4 | Write rejected on policy violation | INSERT returns 403 with policy_class in error |
| 5 | mTLS handshake (internal engines) | `openssl s_client` confirms TLS 1.3; no TLS 1.2 fallback |
| 6 | Nightly CI PENDING_FIRST_RUN flipped | 03-ACCEPTANCE.md §9 Dremio row shows PASS |
